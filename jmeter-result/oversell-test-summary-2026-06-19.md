# TickGo 超卖压测记录

日期：2026-06-19

## 测试目标

- 验证高并发购票场景下，系统是否会出现超卖
- 本轮只关注“成功卖出多少张票”，不关注票具体卖给哪个乘客
- 压测核心判断标准：
  - 最终成功票数是否超过余票数
  - 是否出现重复座位

## 测试方式

- 使用 JMeter 文件：
  - `jmeter/last-seat-100-users.jmx`
  - `jmeter/last-seat-100-users.csv`
- 并发请求数：`100`
- 请求方式：
  - `8082` / `18082` 两个 `ticket-service` 实例交替接收请求
  - 每个请求只携带 1 个 `passengerId`
  - `Synchronizing Timer = 100`，保证 100 个请求尽量同时发出

说明：

- 该轮压测的重点是“库存并发控制”
- 不验证乘客归属和用户权限
- `passengerId` 在本轮中仅作为请求标识使用

## JMeter 结果

| 指标 | 数值 |
| --- | ---: |
| 总请求数 | 100 |
| 平均响应时间 | 1803 ms |
| 最小响应时间 | 543 ms |
| 最大响应时间 | 2679 ms |
| 标准差 | 361.28 ms |
| 错误率 | 97.00% |
| 成功率 | 3.00% |
| 成功请求数 | 3 |
| 失败请求数 | 97 |
| 吞吐量 | 37.3 req/s |
| 平均响应体大小 | 260.7 bytes |

## 结果解读

### 1. 业务结果

- 在 100 个并发请求中，最终只有 `3` 个请求成功
- `97` 个请求失败
- 如果压测当时系统可售余票为 `3` 张，则这组结果与“按库存上限放行、超过库存的请求被拦截”高度一致

### 2. 性能表现

- 平均响应时间为 `1803 ms`
- 最大响应时间为 `2679 ms`
- 没有明显达到 `3s` 锁等待超时上限
- 说明这轮失败请求并不完全表现为“等锁等满 3 秒后失败”，更可能是库存竞争、业务拒绝和锁竞争共同作用的结果

### 3. 对超卖问题的判断

从 JMeter 结果本身看：

- 系统没有把 100 个请求全部放过
- 成功请求数被压缩到 `3`
- 结果趋势上支持“系统具备防止超卖的能力”

但是，JMeter 结果本身还不能单独作为最终结论。  
要真正证明“未超卖”，还需要数据库结果同时满足：

- 最终新增有效票数 = 余票数
- 不存在重复座位记录

## 本轮测试可以说明的结论

### 可以说明

- 100 个并发请求确实进入了购票接口
- 系统在高并发场景下没有无条件全部放行
- 成功请求数明显受库存或并发控制约束
- 如果余票确实为 `3`，则当前结果与“未超卖”相符

### 不能单独说明

- 不能仅凭 JMeter 图证明绝对没有超卖
- 不能仅凭 JMeter 图判断所有失败都来自库存不足
- 不能说明乘客归属、用户权限或购票身份链路是正确的

## 核心指标分析

本轮最有价值的参数如下：

- `# Samples = 100`
  - 说明本轮总共发出了 100 个请求
- `Error % = 97.00%`
  - 说明失败比例很高，只有少量请求成功
  - 结合余票数，可以推断系统是否按库存限制成功请求数
- `Average = 1803 ms`
  - 反映整体平均响应耗时
- `Max = 2679 ms`
  - 反映最慢请求耗时
  - 本轮没有明显达到 3 秒锁超时上限
- `Std. Dev. = 361.28 ms`
  - 反映响应时间离散程度
  - 说明大部分请求耗时集中在相对接近的区间

以下参数本轮参考价值较低：

- `Received KB/sec`
- `Sent KB/sec`
- `Avg. Bytes`

这些更适合网络传输观察，不是超卖判断的核心依据。

## 建议补充的数据库核验

建议配合以下 SQL 判断最终是否真的未超卖：

### 1. 查询最终有效票数

```sql
select count(*) as cnt
from t_ticket
where train_id = 1
  and departure = '北京南'
  and arrival = '南京南'
  and ticket_status in (0, 10);
```

### 2. 查询是否存在重复座位

```sql
select carriage_number, seat_number, count(*) as cnt
from t_ticket
where train_id = 1
  and departure = '北京南'
  and arrival = '南京南'
  and ticket_status in (0, 10)
group by carriage_number, seat_number
having count(*) > 1;
```

## 可直接用于报告的结论

在 100 个并发请求同时竞争余票的场景下，系统最终仅成功处理 `3` 个请求、拒绝 `97` 个请求，平均响应时间 `1803 ms`，吞吐量 `37.3 req/s`。从压测现象看，成功请求数明显受到库存上限约束，未出现“所有请求被同时放行”的情况，结果趋势上支持系统具备防止超卖的能力。若数据库最终确认有效票数等于余票数且不存在重复座位，则可进一步证明该并发售票链路在当前场景下未发生超卖。

## 可直接用于简历的表述

- 使用 JMeter 对双实例 `ticket-service` 进行 100 并发抢票压测，在高并发购票场景下将成功请求控制在库存上限以内，初步验证并发售票链路具备防止超卖能力
- 设计基于 CSV 驱动和同步放行的并发压测模型，模拟 100 个购票请求同时竞争余票，并结合响应时间、失败率和数据库结果分析库存并发控制效果

## Redis 热点 Key 示例

如果需要在总结中说明“使用 Redis 热点 key 做前置拦截”，可以补充下面两条示例语句。

### 1. 无过期时间写入热点 key

```redis
SET ticket:purchase:1:hotkey PROCESSING
```

说明：

- 该写法不会设置过期时间
- 适合演示“将热点 key 直接写入 Redis”
- 如果要做真正幂等抢占，通常会配合 `SETNX` 或 `SET key value NX EX seconds`

### 2. 删除热点 key

```redis
DEL ticket:purchase:1:hotkey
```

在本项目语境下，可将其理解为：

- 写入 key：表示某类重复请求已经被前置标记
- 删除 key：表示请求处理结束后释放该热点拦截标记
